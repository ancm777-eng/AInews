### 항목 3. OpenAI, Codex를 위한 커스텀 Windows 샌드박스 구축

**판정: ⚠️ 사실 확인됨 / 아키텍처 명칭 및 특성 기술 오류**

**✅ 정확한 부분:**
- 
OpenAI가 Codex의 Windows 샌드박스에 대한 기술적 설명을 발표했으며, 이는 개발자 PC에서 코딩 에이전트를 위한 더 엄격한 로컬 설정을 상세히 기술하고 있다.

- 
샌드박스는 Codex가 머신에 무제한 접근 권한 없이 자율적으로 작동할 수 있는 경계를 정의하며, Codex가 로컬 명령을 실행할 때 제한된 환경에서 실행된다.


**❌ 아키텍처 명칭 오류: "elevated sandbox"의 성격 왜곡**

초안은 "elevated sandbox"를 파일 접근과 네트워크 제한에 대한 엄격하고 세분화된 제어를 구현하면서 에이전트가 자율적으로 코드를 실행할 수 있도록 하는 단일 아키텍처처럼 기술한다. 그러나 실제로는:

- 
Codex는 Windows에서 세 가지 방식으로 실행될 수 있다: 더 강력한 elevated 샌드박스를 사용한 네이티브 Windows, 폴백 unelevated 샌드박스를 사용한 네이티브 Windows, 또는 Linux 샌드박스 구현을 사용하는 WSL2 내부 실행이다.

- 
네이티브 Windows 샌드박스 지원은 두 가지 모드를 포함한다: elevated는 전용 낮은 권한 샌드박스 사용자, 파일시스템 권한 경계, 방화벽 규칙, 샌드박스 명령 실행에 필요한 로컬 정책 변경을 사용하는 선호 네이티브 Windows 샌드박스이다.

- 따라서 "elevated sandbox"는 OpenAI가 새로 발명한 단일 완성 솔루션이 아니라 두 가지 네이티브 모드(elevated/unelevated) 중 하나이며, 초안에서 이를 "custom Windows sandbox"라고 단일 명칭으로 부른 것은 구조를 단순화한 것이다.

**❌ 발행일 표현 주의**


OpenAI 엔지니어링 블로그의 "Building a safe, effective sandbox to enable Codex on Windows" 게시글 날짜는 2026년 5월 13일이다.
 초안의 event_date: 2026-05-14와 하루 차이가 있다. KST 기준으로 5월 14일에 해당할 수 있으나, UTC 기준 게시일을 정확히 표기하는 것이 바람직하다.

**🔧 보완 제안:**
- "custom Windows sandbox"의 기술적 정체를 명확히 해야 한다: SID(Security Identifier) + 쓰기 제한 토큰(write-restricted token)을 활용하며, elevated와 unelevated 두 가지 모드가 있음을 명시.
- 
2월에 공개된 GitHub 이슈에 따르면, 초기 샌드박스는 샌드박스 계정이 활성 프로젝트 폴더뿐만 아니라 C:\Users\ 전체 디렉터리에 읽기 접근 권한을 부여해 SSH 워크플로우 손상, DRM 실패 등의 문제를 야기했다.
 이 보안 개선의 맥락을 기술하면 분석 가치가 높아진다.
- "완전히 해결(fundamentally solved)"이라는 표현은 과장이다. 샌드박스는 여전히 진화 중이며, 초안의 Critical View에서 언급한 prompt injection 위험도 OpenAI 자체 문서에서 인정한 사항이다.
